Why developers choose this

Built like a tool you’d

15 MB idle. Not 150.

Most menu bar apps ship a SwiftUI settings window, an NSHostingView, and an entire diffing engine — sitting in memory for a screen you open once a month. KeyMute has zero windows. The entire UI is an NSStatusItem, an NSMenu, and a single NSPanel for the toast. No view hierarchies. No layout solver idling.

~15 MBKeyMute

OS-level mute

Writes kAudioDevicePropertyMute on the system input device via CoreAudio. Same property macOS uses in System Settings. Every app — Zoom, Meet, Teams, Chrome — reads silence instantly.

No Accessibility prompt

Global shortcut uses Carbon RegisterEventHotKey. No Input Monitoring. No “wants to control your computer” dialog. Just Microphone permission on first use.

Invisible to screen share

The toast HUD sets sharingType = .none. You see the “Muted” confirmation. Your audience doesn’t. No awkward popups in your presentation.

700 KB binary

Pure AppKit + CoreAudio. No Electron. No web views. No bundled runtimes. Smaller than most favicons. Launches instantly because there’s nothing to load.

Under the hood

One CoreAudio call. That’s it.

KeyMute writes to the same property macOS reads in System Settings > Sound. The OS propagates the mute to every process. Property listeners keep the icon in sync when something else changes the state.
