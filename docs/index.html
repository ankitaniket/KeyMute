The problem

You’re presenting. Someone
asks a question. You need to mute.

Zoom’s mute is in one corner. Meet’s is in another. Teams hides it behind a toolbar. If you’re sharing your screen, you have to alt-tab back to the call just to click a button — and everyone on the call watches you do it.

You just want one shortcut that works everywhere, regardless of which app is in focus, regardless of which conferencing tool you’re using, regardless of whether you’re mid-presentation or deep in your editor.

⌘⇧0 → mic muted at the OS level → every app hears silence instantly
Why developers choose this

Built like a tool you’d write yourself

15 MB idle. Not 150.

Most “simple” menu bar apps ship a SwiftUI settings window, an NSHostingView, and an entire diffing engine — sitting in memory for a screen you open once a month. That’s 50–150 MB of idle RAM for a mute button. KeyMute has zero windows. The entire UI is an NSStatusItem, an NSMenu, and a single NSPanel for the toast. No view hierarchies. No layout solver. No wasted memory.

50–150 MB Typical menu bar app
~15 MB KeyMute

OS-level mute

Writes kAudioDevicePropertyMute on the system input device via CoreAudio. Same property that System Settings reads. Every app — Zoom, Meet, Teams, Chrome, OBS — immediately reads silence. No per-app integration. No accessibility hacks.

No Accessibility permission

Global shortcut uses Carbon RegisterEventHotKey. No Input Monitoring. No “wants to control your computer” dialog. Just Microphone permission on first use — because that’s what the app actually does.

Invisible to screen share

The toast HUD uses sharingType = .none. You see the “Muted” confirmation. Your audience doesn’t. No awkward popups in your demo or presentation.

700 KB binary

Pure AppKit + CoreAudio. No Electron. No web views. No bundled runtimes. Smaller than most favicons. Launches in under a second, because there’s nothing to load.

Under the hood

One CoreAudio call. That’s it.

KeyMute writes to the same property that macOS reads in System Settings > Sound > Input. The OS propagates the mute to every process reading from the input device. Property listeners keep the icon in sync if something else changes the state externally.
