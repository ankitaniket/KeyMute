No windows means no view hierarchies sitting in memory. No SwiftUI runtime loaded for a settings screen you open once a month. No NSHostingView, no diffing engine, no layout solver idling in the background. The entire UI is an NSStatusItem, an NSMenu, and a single NSPanel for the toast — allocated once, reused forever.

Why this exists

~15 MB idle. Not 150.

Most "simple" menu bar apps ship a full settings window, a SwiftUI runtime, and an NSHostingView — even if you open settings once a month. That's 50-150 MB of idle RAM for a mute button. KeyMute controls everything through the menu bar dropdown. No windows, no hidden view trees, no wasted memory.

50-150 MB Typical menu bar app
~15 MB KeyMute

OS-level mute, not app-level

Writes kAudioDevicePropertyMute on the system input device via CoreAudio. Same property macOS reads in System Settings > Sound. Every app using the mic — Zoom, Meet, Teams, Chrome, OBS — immediately reads silence. No per-app integration, no accessibility hacks, no screen recording permission.

No Accessibility permission

Global shortcut uses Carbon RegisterEventHotKey. No Input Monitoring. No Accessibility prompt. No "KeyMute wants to control your computer" dialog. Just Microphone permission on first use — because that's what the app actually does.

Invisible to screen share

The toast HUD uses sharingType = .none. When you mute during a screen share, you see the confirmation toast. Your audience doesn't. No awkward "Muted" popups in your demo or presentation.

How it works

⌘⇧0  →  AudioObjectSetPropertyData(kAudioDevicePropertyMute, 1)
      →  Zoom: silence  |  Meet: silence  |  Teams: silence  |  Everything: silence

One CoreAudio call. The OS propagates the mute to every process reading from the input device. Property listeners keep the icon in sync if something else changes the mute state externally.
